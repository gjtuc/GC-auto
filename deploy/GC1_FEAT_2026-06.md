## 1. 왜 이번 업데이트가 큰가

GC1은 이제 **Autochro OCR 눈 + 핫스팟 트리거 + Cursor 에이전트/OCR 폴백 + 런 종료 학습**까지 한 사이클로 돌아갑니다.  
한 repo(`gjtuc/GC-auto`)에 GC1·2·3 코드가 같이 있으므로, **GC1 전용 분기는 env·`gc_instance`로만 켜지고** GC2/GC3·데이터 PC 경로는 건드리지 않습니다.

| 구분 | GC1 장비 PC | GC2/GC3 장비 | 은규/차헌 데이터 PC |
|------|-------------|--------------|---------------------|
| 코드 | 동일 repo pull | 동일 repo pull | `data_pc/` 만 사용 |
| 켜지는 기능 | `gc1`, OCR, hotspot agent | `gc2`/`gc3`, 기존 watch | `촉매 반응 계산.py` |
| 출력 폴더 | `Desktop\박은규` | `Desktop\KCH` | `gc-data-pc` / `.cursor` |
| 이번 feat 직접 영향 | **있음** | **없음** (레이아웃·핫스팟 agent) | **없음** |

## 2. 기능 목록 (main에 포함된 GC1 전용)

### 2.1 OCR · Autochro eye (`gc1_runtime/layer3_*`, `gc_autochro.py`)

- `GC1_AUTOCHRO_EYE=1` — 단계마다 OCR 읽기·클릭·검증 (live 기본)
- `GC1_AUTOCHRO_EYE_ADAPT=1` — OCR 실패 시 fallback 후 계속 (학습 모드)
- `GC1_OCR_CASE_STUDY=1` — 실패 시 빨간 박스·탐색·`fail_*.json`
- `GC1_OCR_LEARN=1` — overlay `~/.cursor/gc-ocr-learnings/screen_regions.overlay.json`
- **시료명**: 제어목록 UI/OCR 우선 — `AUTOCHRO_DATA_NAME` 은 fallback 만
- **MTD**: 고정 파일명 `20260629 분석방법.MTD` (하드코딩)

### 2.2 런 종료 closure (`gc1_runtime/layer3_run_closure.py`)

파이프라인 끝에 반드시: 케이스 스터디 수집 → overlay 반영 → `run_journal_latest.json`  
실사용자(은규)에게는 형식적 완료 문구만 (`format_end_user_summary`).

### 2.3 iPhone 핫스팟 → Cursor / OCR (`gc1_runtime/layer0_hotspot_agent.py`)

| 우선순위 | 조건 | 동작 |
|----------|------|------|
| 1 | `CURSOR_API_KEY` + cursor-sdk | `Agent.prompt("동작해")` — IDE 꺼져도 SDK 가능 |
| 2 | Cursor 불가·실패 | Python OCR **force** (`run_force_once`) |
| 쿨다운 | `GC1_HOTSPOT_RECONNECT_MIN_SEC=1800` (30분) | 동일 세션 중복 방지 |

`GC1_HOTSPOT_CURSOR_AGENT=0` → watch 가 예전처럼 직접 pipeline (레거시).

### 2.4 데이터 폴더 분리 (`gc_profiles.gc_runtime_dir`) — **이번 feat 커밋 핵심**

```
Desktop\박은규\
  *.xlsx, *.pdf     ← 은규·운영자가 보는 데이터만
  _GC자동화\        ← env·watch·로그·bat·Cursor 연동
```

- `migrate_gc1_runtime_layout()` — `gc_automation.py` 시작 시 루트에 남은 자동화 파일 자동 정리
- 상세: `deploy/GC1_DATA_FOLDER_LAYOUT.md`

### 2.5 운영 도구

| 명령 | 용도 |
|------|------|
| `python scripts/verify_gc1_run_trace.py` | OCR 폴백·저널·핫스팟 로그 한 화면 |
| `python scripts/gc1_show_last_run.py` | 마지막 런 요약 |
| `python gc1_runtime/layer0_hotspot_agent.py --status` | SDK·쿨다운·Wi-Fi |
| `gc1_바탕화면_바로가기.bat` | bat → `_GC자동화\` 에 생성 |

## 4. PC별 pull · 병합 절차 (충돌 방지)

### 4.1 GC1 장비 PC (지금 PC)

```bat
gc_git_pull.bat
gc_stop_watch.bat
gc_start_watch.bat
```

- 첫 pull 후 `gc_automation.py` 1회 실행 → `_GC자동화` 로 파일 정리 메시지 확인
- `python gc_automation.py --show-profile` → `자동화 폴더: ...\_GC자동화` 표시

### 4.2 GC2/GC3 장비 PC (차헌)

```bat
gc_git_pull.bat
```

- **변경 없음**: `gc_runtime_dir()` 은 `gc1` 일 때만 `_GC자동화` 사용. KCH 레이아웃 동일.
- 기존 `Desktop\KCH\gc_automation.env` 그대로.

### 4.3 은규 PC / 차헌 데이터 PC

```bat
gc_git_pull.bat
```

- `gc_automation.py` **실행하지 않음** — `data_pc/촉매 반응 계산.py` 파이프라인만.
- 원격 `feat/data-pc-origin` 과 병합 시: **GC1 변경은 `gc1_runtime/`, `gc_watch.py` 의 GC1 분기** — 데이터 PC 스크립트와 파일 겹침 거의 없음.

### 4.4 Git 브랜치 전략 (권장)

1. `feat/gc1-2026-06-runtime` — GC1 레이아웃·문서·마무리 패치
2. `main` merge (`--no-ff` 권장 — feat 경계 명확)
3. 차헌 PC 대규모 작업 브랜치는 **최신 `main` pull 후** rebase 또는 merge
4. 충돌 나면 보통 `gc_profiles.py`, `gc_watch.py` — **GC1 블록만** 유지, GC2/GC3 블록은 차헌 쪽 유지
